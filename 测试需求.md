## 1. 背景与目标

### 1.1 当前痛点

目前手头**没有相机、喷阀阵列、编码器**等硬件，但希望在不打板的情况下，用真实日志 + 真实采图回放整条核心管线，验证：

- 采图 → 软件节流 → YOLO 推理 → 跨帧 mask‑IoU 合并 → 喷阀脉冲预计算

**链路是否在功能上闭环、关键时序是否正确**。这相当于在测试机上模拟"硬件挂着皮带在跑"的效果，但所有外设全部用回放数据替代。

### 1.2 一句话目标

> 用 `/Users/fangjin/Downloads/saveRawPic/`（与测试机 `/home/pi/offline_sim_data/saveRawPic/` 内容一致）下的：
>
> - **每 2 帧保留 1 帧**的相机原始 JPG 图片序列；
> - 同目录下的 `log.txt`（包含编码器 `线速度：Xm/min`、相机 `--取图--<时间戳> 第N次`、识别 `--开始识别/识别完成--<时间戳> 第N次` 等事件）；
>
> 替代真实相机和真实编码器，驱动现有 `YoloWorker → TrackerWorker → Dispatcher` 链路，把 Dispatcher 即将下发给 `boardControl` 的 `enqueuePulses / cancelPulses` 命令**捕获到日志/CSV**中以供检查，在**测试机 RK3588** 上完整跑通核心流程。

### 1.3 期望产出

1. 一个独立的离线仿真小工具（推荐放在 `tests/offline_sim/` 或 `tools/offline_sim/`），通过 `qmake` 子工程生成可执行文件。
2. 该工具在测试机 `pi@192.168.50.215:/home/pi/code/QINGLV/` 上能 `make && ./offline_sim` 跑完两段 session（11:05–11:11、11:26–11:35）。
3. 仿真结束后落盘以下产物到 `offline_sim_outputs/<run_ts>/`：
   - `events.csv`：仿真喂入的每一帧采图事件 + 编码器速度样本（绝对仿真时间戳 / 序号 / 速度 / 文件名）。
   - `valve_commands.csv`：捕获到的 `enqueuePulses` 和 `cancelPulses` 全量记录（trackId / boardId / channelMask / tOpen / tClose）。
   - `sort_tasks.csv`：Tracker 触发的所有 `SortTask`（trackId / classId / tCapture / bbox / area）。
   - `warnings.log`：链路中所有 `warning(...)` 信号文本汇总。
   - **`annotated_frames/<seq>.jpg`**（业务方要求）：**真正经过 YOLO 推理的每一帧（约 N_yolo ≈ 52 张）**叠加以下元素后落盘，便于人工验收：
     - 检测 mask 半透明覆盖（每个 detection 一个）
     - **bbox 框**
     - **每个 mask 同时画两个类别 + 置信度**（业务方明确：原始识别和最终标记都要看到）：
       - `yolo=<det.classId 中文名> conf=<det.confidence>` —— 该 det 当帧 YOLO 给出的**原始识别**类别
       - `best=<bestClassId 中文名>` —— 该 det 所属物体（active track 或 ghost）此刻的**最终标记**类别（永远有值，取值规则见 §3.6 R2 表中的 `bestClassId` 注释）
     - **跨帧合并后的 `trackId`**：合并到 active 或新建 track 时画该 id；进 ghost 池时画 `T?`
     - **不画** `classAreaAcc[bestClass]` 的具体面积值（业务方明确不需要，避免标签太挤）
     - 推荐文字布局：每个 mask 框上方两行，行 1 = `T<trackId> | best=<...>`，行 2 = `yolo=<...> conf=<...>`；`trackId=-1` 时行 1 = `T? | best=<...>`
   - `frame_assoc.csv`：每帧每个 detection 与 trackId 的绑定，列名建议：
     ```
     seq, tCaptureMs, detIndex, trackId, bestClassId, detClassId,
     confidence, bbox_x, bbox_y, bbox_w, bbox_h,
     suppressedByGhost, isNewTrack, firstFrameGhost
     ```
     是 `annotated_frames/` 的数据底稿，也方便 grep（如 `grep ",7," frame_assoc.csv | awk -F',' '$4==7'` 找 trackId=7 的所有帧）。
   - `summary.md`：本次跑的统计指标（见 §3.4）。

### 2.4 数据消费规则（约束）

- **数据集语义**（业务方第三批补充澄清）：`saveRawPic/` 里的偶数 seq JPG 序列**等价于"相机原始流"**（旧分支侧硬件帧节流后的存档），**不**等价于"已经过软件节流可以直接送 YOLO"的帧序列。`OfflineCameraDriver` 必须像生产路径上的 `CameraWorker` 一样，对这条原始流再做一次"`softFps=2` 软节流 + 单元素反压"，结果只有大约 **2 fps × 时间窗** 的帧最终进 `YoloWorker::onFrame`。
  - 反压口径与生产一致：`m_inFlight ≥ 1` 时跳过该 tick，并累计 `m_droppedSinceWarn`（与 `CameraWorker` 一致），≥5 且距上次告警 ≥2 s 时打 `[WARN]` 到 `warnings.log`。
  - 不重写节流算法 —— 直接复用 `CameraWorker` 同款逻辑（`m_intervalMs = 1000/softFps`、`m_inFlight` 计数、`onFrameConsumed` 解锁），实现层面 `OfflineCameraDriver` 等于"`CameraWorker` 把 MV_CC_GetImageBuffer 换成 cv::imread"。
- **首条速度日志之前的数据全部跳过**（业务方明确要求）：`log.txt` 中 Session #1 第一条 `[MainWindow] --速度--` 出现在 `2026-04-30 11:05:44.171`（线速度 10 m/min），Session #2 同理取该 session 内第一条速度行。
  - 仿真起点 `t0_log` 取该速度时间戳本身，**不取**该 session 第一条 `--取图--`。
  - 时间早于 `t0_log` 的 `--取图--` 事件**根本不进 `OfflineCameraDriver` 的待发列表**（即使对应图片仍在 `saveRawPic/` 目录中）。
  - 时间早于 `t0_log` 的 `--速度--` 事件……不存在，按定义 `t0_log` 自身就是首速度。
  - 这与生产路径"`TrackerWorker::onFrameInferred` 没有有效速度时退化 nominal"的口径保持一致：宁可少跑几帧，不要在没有速度的状态下做位移外推。
- **`tCaptureMs` 的取法**：仿真启动时刻 `pipeline::nowMs()` 为新原点 `t0_sim`，任意事件的仿真时间戳 = `t0_sim + (t_log - t0_log)`，其中 `t0_log` 见上一条。
- **session 切换**：本期默认仿真 **Session #1 单段**，Session #2 留给后续轮次（避免一次跑太久 + 不掩盖 Session 之间的状态污染问题）。Session 选项作为 `--session=1|2|all` 命令行参数。
- **本期默认仿真子集（业务方要求小批量）**：seq ∈ **`[2, 334]`** 的偶数帧 = `saveRawPic/` 里 167 张 JPG（"相机原始流"）。
  - **首速度门槛**之后实际进入 `OfflineCameraDriver` 待发列表的"原始流" = **162 张**（`seq=12` 11:05:44.233 起 → `seq=334` 11:06:10.239 止，时间窗约 **26.0 秒**；`seq=10` 在 11:05:44.079，仍早于首速度 11:05:44.171 故被剔除）。Session #2 同样 162 张（`seq=12` 11:26:20.326 起，t0_log=11:26:20.238）。该值已用 M0.5 `sanity_check` 在真实 `log.txt` 上端到端核实。
  - **再叠 `softFps=2` 软节流后真正过 YOLO 的帧**预计 ≈ 26.0 s × 2 fps ≈ **52 帧**（实际值由仿真跑出，受反压实时表现影响，可能偏低 ~5 帧）。后续文档里 `annotated_frames/` 张数、`frame_assoc.csv` 行数都按"过 YOLO 的实际帧数 N_yolo"算，**不**按 162。
  - 这个窗口横跨"皮带启动加速段（前 ~6 秒，0→31 m/min）"+"31 m/min 稳定段"，覆盖速度变化场景，足够触发 AC8（速度 Δ > 20% 重算）。
  - §3.5 标定窗口（seq=20…70）落在该子集内，但**不能复用仿真主流程的 detection 输出**（仿真主流程会被节流跳过大量帧，标定需要的 26 帧大概率被丢）。标定要单独跑一次"全帧无节流"模式（详见 §3.5.2 修订版）。
  - CLI：`--seq-from=2 --seq-to=334`；不传则跑整个 session；可加 `--no-throttle` 跳过软节流（仅供 §3.5 标定使用，**不在常规仿真里用**）。

### 3.4 通过标准（Definition of Done）

仿真跑完以下都成立才算"过"：

1. 进程零 crash、零 `assert` / 段错误。
2. `warnings.log` 里无 `[ERROR]`，仅允许少量已知 `[WARN]`（如 `object_too_small`，需在 `summary.md` 解释每一条原因）。
3. 链路无内存泄露：跑完一段 session 后内存与刚启动时差 < 50 MB（按需求 §6.1）。
4. **`annotated_frames/` 完整性**：进入 YOLO 的每一帧（实际数量 N_yolo ≈ 52，由仿真跑出，与 `frame_assoc.csv` 行的"unique tCaptureMs 数量"对得上）都必须落 1 张标注图；标注图上每个 mask 必须能看到 4 个字段：
   - `trackId`（被 ghost 抑制的 det 标 `T?`）
   - `bestClassId`（最终标记的类别，永远有值；规则见 §3.6 R2 注释）
   - `det.classId`（YOLO 当帧的原始识别类别）
   - `confidence`

   业务方人工抽查 ≥ 10 张，确认 `mask × trackId × bestClassId × detClassId × confidence` **五元组**与肉眼判断一致；特别关注：
   - 同一 trackId 的多帧之间 `bestClassId` 是否稳定（说明合并正确）；
   - `bestClassId ≠ det.classId` 的帧（说明 YOLO 当帧抖动但累计类别保持）能否在多帧中合理观察到。
5. **业务正确性抽查（依赖 #4）**：
   - 随机挑 ≥ 5 条 `SortTask`，按 `trackId` 反查 `annotated_frames/`，确认该物体在多帧中**始终是同一 trackId**（即 R1 修复有效，没被拆成两条）。
   - AC4：选 ≥ 2 个不同 trackId 的物体，确认各自只触发 1 次 SortTask。
   - AC15：Session 起点附近的 annotated 帧上首批 det 的 trackId 应为 `T?`（进 ghost 池）而非新 trackId。
   - AC17：找 ≥ 1 个"被分拣后又出现在视野中"的物体，确认其后续帧标的还是原 trackId 或 `T?`，**不是**新 trackId。
6. AC4 / AC5 / AC15 / AC17 各至少有一个真实样例命中并被记录到 `summary.md`（直接贴对应 annotated 帧的相对路径作为证据）。
7. **YOLO 推理耗时量级与真机一致**（M1 拍板的"参考校验"，非阻塞）：仿真 `YoloWorker` 落出的 `tInferDoneMs - tCaptureMs` 平均值应落在 **`[200, 600] ms`** 之间，与真机 `log.txt` 实测真值 **avg=349 ms / range=[310, 406] ms / 1981 对配对成功**（来自 M1 `sanity_check`）量级一致。偏差超过 ±50% 时记入 `summary.md` 提示"模型/阈值或环境差异"。

### 3.5 角速度 ↔ 线速度系数标定（M1 前置任务）

业务方反馈：`log.txt` 里 `线速度：X m/min` 字段**不准**，`转速：Y r/min`（角速度）相对准确。需要先用一段**视野内只出现单个物体**的连续帧反推真实"`r/min` → `mm/ms`"换算系数 `K`，标定后 `OfflineEncoderDriver` 用：

```
speedMmPerMs = K × rpm   (mm/ms = K × r/min, K 单位 mm·min/(ms·r))
```

而**不是**用 log 里的 `线速度` 字段。

#### 3.5.1 标定数据集

| 项 | 值 |
|---|---|
| 时间窗 | `2026-04-30 11:05:43.450` 至 `2026-04-30 11:05:50.355`（约 6.905 秒） |
| 起始帧 | `img_20260430_110543_450.jpg_20.jpg` |
| 结束帧 | `img_20260430_110550_355.jpg_70.jpg` |
| 帧数 | seq 从 20 到 70，步长 2，共 26 帧（保留下来的偶数帧） |
| 视野内物体数 | 仅 1 个（业务方确认） |
| 该窗口内的 r/min 序列 | 由 log 抽出对应 `--速度--` 行，按时间逐条平均 |

#### 3.5.2 标定流程（修订：独立的"无节流"模式）

由于常规仿真的 `softFps=2` 软节流会把 26 帧丢成 ~13 帧（且哪些被丢由反压实时表现决定，无法重现），**标定必须单独跑一次"无节流"模式**：把 26 张全部强制送进 `YoloWorker` 取 mask，再做拟合。

具体步骤（建议放 `tests/offline_sim/calibrate_rpm_to_speed.cpp`，**独立可执行**，不进主仿真程序，不进生产路径）：

1. CLI: `./calibrate_rpm_to_speed --data-dir=/home/pi/offline_sim_data/saveRawPic --seq-from=20 --seq-to=70`。
2. 直接构造一个 `YoloWorker`（同生产代码 `preloadModel + sessionStart`），按 seq 顺序串行调用 `onFrame`，**绕过 `OfflineCameraDriver` 的节流逻辑**（即每帧都送，不管 inFlight）。
3. 收到 26 个 `detectedFrameReady` 后逐帧落到 `calib_detections.json`，每行：`{seq, tCaptureMs, objs:[{classId, conf, bboxPx, centerPx}]}`。每帧只允许 0 或 1 个目标（>1 时报错，让人工换窗口）。
4. 拟合：
   - `y_mm = y_px × 1520 / imgH_px`（取 `centerPx.y`，像素重心已由 `postprocess_ex` 算好）。
   - 用最小二乘拟合 `y_mm` 关于 `(t_log - t0_log)`（单位秒）的直线斜率 `slope_mm_per_s`。换算 `v_real_mm_per_ms = slope_mm_per_s / 1000`。
5. `rpm_mean` = 该窗口内 log `转速` 字段的均值（含跨多条速度行）。
6. 标定系数 `K = v_real_mm_per_ms / rpm_mean`，单位 `mm·min/(ms·r)`。
7. 输出到 `offline_sim_outputs/calibration/<run_ts>/`：
   - `calib_detections.json`
   - `calib_fit.csv`（26 行，每行 `seq, t_relative_s, y_px, y_mm, residual_mm`）
   - `calib_result.txt`（`K`、R²、残差最大值、所用 rpm_mean）
8. `K` 数值人工 commit 到测试分支某个 ini（暂命名 `tests/offline_sim/offline_sim.ini`），主仿真程序在启动时读取；**不污染 `config.ini`**，不污染 `RuntimeConfig`。

#### 3.5.3 风险与守门

- 若 `R² < 0.95` 或物体跨视野时被截断（首末帧只看到一半物体导致质心跳变），标定结果不可用 —— 需要换一段窗口或人工筛掉边界帧。
- 若该 26 帧内**编码器 raw 也明显抖动**（log 里 `转速` 在 19/20 之间反复跳），用均值即可，不需要做时变 K 的拟合（皮带是匀速段）。
- 标定完成前 `OfflineEncoderDriver` 不允许跑端到端仿真（防止用错系数得到一份误导性的 `valve_commands.csv`）。

#### R1：`TrackerWorker` 帧间 mask 合并仅靠 IoU，对"残→全"不鲁棒（业务方指出）

| 项 | 内容 |
|---|---|
| 影响文件 | `src/pipeline/tracker_worker.cpp`（`onFrameInferred` 中关联候选打分那一段；当前仅算 `maskIoU`） |
| 业务场景 | 同一物体在前一帧 a 处于视野边界、mask 残缺；下一帧 b 已完整进入视野、mask 完整。a 是 b 的子集（或近似子集），但因为 \|A\| ≪ \|A∪B\|，IoU 会被压低，跨阈值失败 → 认为是新物体，触发误新建 track 而非合并 |

**`AnnotatedFrameSink` 实现要点**：

- 输入两路 Qt 信号（均 QueuedConnection 到自身线程）：
  - `YoloWorker::detectedFrameReady(DetectedFrame)` —— 每帧的 det 列表（带 mask/bbox/classId/conf）。
  - `TrackerWorker::frameAnnotationReady(tCaptureMs, QVector<DetTrackBinding>)`（**R2 新信号**）—— 每帧的 detIndex → trackId 映射。
- 内部用 `QHash<qint64 tCaptureMs, PendingFrame>` 缓存，两路信号都到齐才出图（先到的等后到的）。
- 原图来源：`OfflineCameraDriver` 在 emit `frameReadySig` 时，把 `(tCaptureMs → originalFilePath)` 写到一个进程内可见的小 map（线程安全），`AnnotatedFrameSink` 用 `cv::imread(filePath)` 加载原图。这样不必把整张 `QImage` 在两路信号里都拷贝一份。
- 落盘命名：`annotated_frames/<seq>_<tCaptureMs_relative>.jpg`，`<seq>` 是仿真窗口内的帧号（与 `frame_assoc.csv` 行号一致），便于按顺序翻页。
- 失败容忍：若 `detectedFrameReady` 来了但 `frameAnnotationReady` 在 5 秒内没到（理论上不可能，留个保险），把 binding 视为全 `T?` 出图并 `[WARN]` 入 `warnings.log`。

**关键不变量**：

- 不修改 `YoloWorker / TrackerWorker / Dispatcher / pipeline_types.h / runtime_config.h` 的接口或行为；只**包一层**。

### 4.3 时间轴对齐策略与节流伪代码

```
// 与 §2.4 一致:t0_log = 该 session 内第一条 [MainWindow] --速度-- 的真实时间戳
t0_log  = first_velocity_event.t_log      // Session #1 = 11:05:44.171
t0_sim  = sim.start() 调用瞬间的 pipeline::nowMs()

对每个 log 事件 e (含 --取图-- 和 --速度--, 仅 e.t_log >= t0_log 的):
    delta_ms = e.t_log - t0_log           // 真值偏移
    fire_at  = t0_sim + delta_ms          // 仿真"原始流"应该出现的时间
    QTimer::singleShot(max(0, fire_at - pipeline::nowMs()), this, [=]{ on_raw_frame_arrived(e); });

注意:首条速度事件本身在 t0_log 时刻 emit (delta_ms=0),
     是仿真链路收到的第一条 SpeedSample;紧随其后才是首批"原始流"取图事件。
```

**OfflineCameraDriver 的 softFps 节流 + 反压**（与生产 `CameraWorker` 同款）：

```cpp
// 状态
qint64 m_lastEmitSimMs   = -1;
int    m_intervalMs      = 1000 / cfg.softFps;  // 默认 500
std::atomic<int> m_inFlight{0};
int    m_droppedSinceWarn = 0;
qint64 m_lastWarnMs       = -1;

void on_raw_frame_arrived(const RawFrame& e) {
    qint64 now = pipeline::nowMs();
    bool throttle_skip = (m_lastEmitSimMs >= 0) && (now - m_lastEmitSimMs < m_intervalMs);
    bool backpressure_skip = (m_inFlight.load() >= 1);
    if (throttle_skip || backpressure_skip) {
        if (backpressure_skip) {
            m_droppedSinceWarn++;
            if (m_droppedSinceWarn >= 5 && now - m_lastWarnMs >= 2000) {
                emit warning(QString("OfflineCameraDriver: 已累计丢帧 %1").arg(m_droppedSinceWarn));
                m_lastWarnMs = now;
            }
        }
        return;
    }
    QImage img = QImage(e.filePath);
    qint64 tCapture = now;          // 与生产一致：tCaptureMs 在 emit 这一刻取
    m_inFlight++;
    m_lastEmitSimMs = now;
    emit frameReadySig(img, tCapture, e.fileName);
}
void onFrameConsumed() { m_inFlight--; }

// 标定模式：cfg.softFps=∞ + m_inFlight 永远视为 0，相当于全帧串行送，无丢
```

> 风险：测试机 NPU 推理 ~370 ms/帧，软节流 500 ms 间隔下绰绰有余、不会触发反压告警；如真触发就是新分支推理性能回归，应作为 R3 登记 §7。

### 4.4 SortTask 真值人工抽查工具

人工验收的主入口是 §1.3 第 #4 项的 `annotated_frames/` —— 每张图带 mask + class 中文名 + **trackId** + bbox，由 `AnnotatedFrameSink`（§4.1）负责合成。

抽查路径建议：

1. 拿 `sort_tasks.csv` 的 `trackId` 列。
2. `grep "T<trackId>" frame_assoc.csv` 看该物体在哪几帧出现过。
3. 按这几个 `<seq>` 翻 `annotated_frames/<seq>_*.jpg`，肉眼连看 mask 形状是否一致、class 是否稳定、trackId 是否始终是同一个。
4. 如发现同一物体被拆成两条 trackId（应该被 R1 修掉的场景），登记新 bug 走 §5.2 流程。

**关掉 `YoloWorker` 自身落盘**：仿真里 `cfg.saveRaw = false; cfg.saveResult = false;`，避免 `YoloWorker::draw_results_ex` 又落一份**没有 trackId 的** `result_*.jpg`，与 `annotated_frames/` 重复并误导验收。
